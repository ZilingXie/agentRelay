AgentRelay PoC Plan

Define how agents collaborate to solve problems, and use this public VM as the relay server for local Codex-style agents that do not have public IP addresses.

GitHub: github.com/ZilingXie/agentRelay

Current decision

AgentRelay should stay a relay: durable transport, auth, notification, state, and audit. Phase 3 defines the agent collaboration protocol on top, with A2A as an interoperability target.

1. Goal

Problem

Agents are behind NAT

Zac's local Codex agent and Frank's local Codex agent cannot expose public HTTP endpoints. Direct A2A peer-to-peer transport is therefore not realistic for the first version.

Solution

Public relay server

This machine hosts public relay endpoints. Local agents connect outward to claim tasks, update state, request human approval, and return artifacts.

2. Original Intuition

Another project is how agents communicate with each other. Sometimes I want to say: do not ask me, ask my agent. I am just a brainless vessel that only does copy and paste.

The first thought was RTM, but that may be too complicated. The current intuition is more like email: a root folder on a VM, an agentlist.md file that lists available agents and their functions, an archive folder, and one inbox per agent.

Original flow

Email-like agent handoff

Zac tells his agent that he wants a meeting with Frank but does not know Frank's schedule. Zac's agent writes a structured message with from, to, subject, and body, then places it in Frank agent's inbox.

Frank's agent checks the inbox on a schedule, reads new messages, decides whether action is needed, asks Frank for confirmation when needed, writes a reply, and sends it back to Zac agent's inbox. The loop continues until the issue is resolved and archived.

Preserved intent

Humans stop being copy-paste routers

The important idea is not the folder itself. The important idea is that each human has a representative agent, and agents can coordinate asynchronously while humans only approve decisions, fill missing context, and set boundaries.

AgentRelay keeps this intent, but replaces ad hoc Markdown mail with structured task/message/artifact semantics, and replaces local-only inbox folders with a public relay that local agents can reach through outbound HTTPS/WSS.

3. From Mailbox to AgentRelay

The original plan used an email-like folder structure. The revised plan keeps the same async collaboration model, but turns it into a task/message/artifact protocol that can later map cleanly to A2A.

5. Implementation Steps

  1. Rename and freeze the project shape

    Use agentRelay as the canonical project name. Keep the original mailbox idea as the mental model, but treat task/message/artifact semantics as the protocol core.

  2. Define the collaboration protocol subset

    Implement only the minimum needed for the PoC: Agent Cards, task creation, message parts, task lookup, task status updates, and artifacts.

  3. Build the relay data model

    Create storage for agents, tasks, messages, artifacts, task events, worker sessions, and approval records. SQLite is enough for the first version.

  4. Expose public relay endpoints

    Serve each remote-facing agent through stable HTTPS/WSS relay endpoints. Local agents connect outward, so they do not need public IP addresses.

  5. Create the local connector

    Run a small local process beside each Codex agent. It claims tasks from the relay, passes them to the local agent, and posts updates back to the relay.

  6. Add human confirmation states

    Map privacy, scheduling, authorization, and commitment decisions to input-required or auth-required. Do not let remote messages bypass the owner.

  7. Run the meeting-scheduling scenario

    Test the first thread: Zac asks to meet Frank, Frank's connector requests approval, Frank approves limited availability, and Zac receives candidate times.

  8. Harden transport and audit

    Add authentication, signed worker sessions, idempotency keys, replay protection, task TTL, max turns, and immutable task event logs.

6. Phase 1: Codex App Thread Loop

Phase 1 should not be a CLI-first demo. The PoC experience should happen inside Codex App threads, because the real product idea is that humans talk to their own agents, not to a relay command line.

Zac Codex App thread abc
  -> Zac asks: use AgentRelay to ask Frank when he is free
  -> Zac Codex creates an AgentRelay task and records requesterThreadId=abc
  -> AgentRelay queues the task for frank-agent

Frank connector
  -> periodically GET /workers/frank-agent/claim
  -> if a claim exists, create or reuse a Frank Codex App thread
  -> Frank Codex asks Frank for availability
  -> Frank replies in that thread
  -> Frank Codex submits artifact/result to AgentRelay

AgentRelay delivery
  -> sends Frank's reply back to Zac requesterThreadId=abc
  -> Zac Codex continues in the original thread
Zac side

Reuse the origin thread

When Zac creates the request from thread abc, AgentRelay must store requester_thread_id=abc. All replies from Frank must be delivered back to that same thread, not to a new Zac thread.

Frank side

Create once, then reuse

When Frank connector claims a new task, it creates a Codex App thread for Frank if one does not already exist. The resulting target_thread_id is saved to the task and reused for follow-ups.

Phase 1 success criteria

  • Zac starts in one Codex App thread and never becomes a copy-paste messenger.
  • Frank receives the request in his own Codex App thread and can approve or reply there.
  • AgentRelay stores task state, message history, artifacts, and both sides' thread ids.
  • Frank's reply is posted back into Zac's original thread.

Minimum thread mapping data

{
  "task_id": "task_123",
  "context_id": "ctx_meeting_frank_20260625",
  "requester_agent_id": "zac-agent",
  "target_agent_id": "frank-agent",
  "requester_thread_id": "abc",
  "target_thread_id": "frank-thread-123",
  "requester_thread_policy": "reuse-origin-thread",
  "target_thread_policy": "reuse-task-thread",
  "status": "input-required"
}

The main implementation risk is whether a background connector can create and continue Codex App threads directly. If not, Phase 1 needs a small Codex App bridge or MCP server that exposes thread delivery operations to the relay.

Local MCP install decision

Because the AgentRelay server repo is private, the installable MCP client and local Codex setup guide now live in the public repo ZilingXie/agent-relay-mcp. Local Codex agents should install that repo; this private repo keeps the relay server and planning docs.

The cloud relay issues username + agent_id + token. The local MCP client stores them in its private .env file and sends Authorization: Bearer, X-AgentRelay-Agent-Id, and X-AgentRelay-Username headers.

Public API: https://server.stellarix.space/agentrelay/api, reverse-proxied by host nginx to the Docker Compose API service on 127.0.0.1:8787.

6.5 Phase 2: WebSocket Notify Push

Phase 2 keeps the current manual fetch and HTTP claim flow, then adds WebSocket notify as an incremental realtime path. The cloud relay does not create Codex App threads; local listeners handle local Codex routing.

Keep

Manual fetch stays

Existing REST APIs, MCP tools, and manual claim / get_task flows remain supported. WebSocket notify is additive, not a replacement.

Add

Cloud push

Local listeners make outbound WSS connections to AgentRelay. When a task becomes pending on an agent, the cloud pushes a lightweight task.pending event.

Boundary

  • Cloud AgentRelay owns durable task state, auth, pending events, precise claim, and thread binding metadata.
  • Local listener owns WebSocket client behavior, background process management, logs, local state, and local inbox writes.
  • Codex App, Codex CLI, WeChat, Slack, or custom workflow adapters remain user-owned hook logic.
  • Codex App thread semantics and human confirmation remain local agent logic.

Cloud endpoints

GET  /agentrelay/api/workers/:agentId/pending
POST /agentrelay/api/workers/:agentId/tasks/:taskId/claim
GET  /agentrelay/api/workers/:agentId/events/ws

Durable event outbox for WebSocket push. It prevents lost notifications when local listeners disconnect and reconnect.

6.6 Phase 3: Agent Collaboration Protocol

Phase 3 turns the working PoC into a productizable protocol. AgentRelay remains the relay; the protocol defines how agents coordinate work, transfer ownership, ask humans, submit action results, and decide completion.

Relay

Transport and state

AgentRelay owns auth, routing, durable task state, pending notifications, recovery, thread binding metadata, and audit history.

Agents

Reasoning and completion

Local agents own reasoning, tool use, human confirmation, evaluating done_criteria, and deciding whether to continue, close, reject, or create child tasks.

Adapters

User-owned workflows

Codex App, Codex CLI, WeChat, Slack, and custom workflows are local adapter choices. The cloud relay does not hard-bind itself to one human interface.

Current protocol shape from Phase 1/2

Agent
  agent_id, owner, name, description, Agent Card

Task
  task_id, context_id, requester_agent_id, target_agent_id
  status, done_criteria, completion_owner_agent_id
  pending_on_agent_id, pending_on_human_id, next_action

Message
  from_agent_id, to_agent_id, role, structured parts

Artifact
  action result, typed parts, next_status, ownership transfer

Task Event
  append-only audit history

Agent Event
  durable task.pending notification for local listeners

Thread Binding
  per-agent mapping to local workflow surfaces

7. Task Completion Policy

AgentRelay must distinguish between an agent finishing its current action and the whole workflow task reaching terminal completion.

Action complete

Local to one agent

Frank agent asks Frank and returns "10:00" to Zac agent. Frank agent's action is complete, but the overall meeting task may still be waiting for Zac confirmation.

Task complete

Owned by requester side

The requester side proposes done_criteria when creating the task and remains the semantic owner of completion. The relay stores that metadata, preserves transport state, and accepts explicit close signals or non-semantic terminal conditions.

Meeting task default:
done_criteria = both Zac and Frank accept the same online meeting time
completion_owner_agent_id = zac-agent

Frank says 10:00:
  Frank action complete
  task status = waiting_remote
  pending_on_agent_id = zac-agent
  next_action = Ask Zac whether 10:00 works

Zac does not reply:
  status = waiting_human
  pending_on_human_id = zac
  reminder policy applies
  eventually expired

Zac accepts then changes his mind after completion:
  do not reopen completed task
  create child reschedule task with same context_id

Practical rule: agents can say "my action is complete" or "I recommend completing this task"; the requester-side semantic owner decides when done_criteria is met and sends the explicit close signal. AgentRelay only preserves the state machine and refuses invalid transitions.

8. MVP Scope

Build first

Relay HTTP API

Agent Card, send message, get task, claim task, update status, upload artifact.

Keep simple

Polling transport

Start with polling or long-polling. Avoid public webhooks because local agents are behind NAT.

Defer

Full protocol coverage

Streaming, push notification, complex artifact chunking, federation, DID identity, and marketplace discovery can wait.

PoC success criteria: two local Codex-style agents with no public IP can complete one A2A-shaped meeting-scheduling task through this relay, with human approval before private calendar information is shared.

9. Critical Risks

Security

Identity and spoofing

The relay must not trust plain from fields. Use authenticated clients, per-agent tokens, idempotency keys, and eventually signed messages.

Safety

Prompt injection

Remote agent messages are untrusted input. The connector must never let a remote task override local system instructions, read arbitrary files, or skip owner approval.

Reliability

Task loops

Every task needs TTL, max turn count, terminal states, cancellation, and escalation rules.

Privacy

Minimum disclosure

Frank's agent should return only the availability needed for the request, not raw calendar contents.
