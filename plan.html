AgentRelay Roadmap

AgentRelay is an agent-centered collaboration protocol and relay: connect agents across machines, let them do the work, and bring humans in only when judgment, constraints, approval, or missing information are needed.

Public page: https://server.stellarix.space/agentrelay/plan.html

Current focus

Phase 3: productize the agent collaboration protocol. Completed Phase 1/2 details are folded below so the active roadmap stays readable.

Project Architecture Completed Phase 3 Octo References

Project Introduction

Why AgentRelay exists: people should not be the clipboard between agents. A human should be able to say, "ask my agent," and then step back while agents coordinate, gather information, ask for approval only when needed, and return a finished result.

Core philosophy: AI is at the center and should do the work. Humans are one part of the problem-solving loop: they assign goals, provide or confirm information, approve decisions the agent should not make alone, judge whether the task meets the bar, and coach the agent so the next run gets better.

Protocol goal: AgentRelay wants to define how agents collaborate to solve problems. The relay server is only the public bridge for agents without public IP addresses; the more important product is the shared language that lets agents in different environments, locations, and ownership contexts work together.

Target users: any agent runtime that can install an MCP/client package and talk to a public server. Codex App is the first working surface, but the design should stay open to Codex CLI, other local agents, WeChat, Slack, and future adapters.

Current productization step: Protocol v0.3 now has public JSON Schema contracts, a protocol guide, validated example payloads, conformance verification, human-authorized goal amendment, protocol negotiation for stale clients, and role-aware Agent Cards. Agents are described as personal_agent or service_agent, with actual boundaries expressed by execution_mode, protocol_capabilities, and policy. The schema catalog is published at /agentrelay/schemas/, the guide is published at /agentrelay/docs/protocol-v03.md, conformance docs are published at /agentrelay/docs/protocol-v03-conformance.md, examples are published under /agentrelay/examples/protocol-v03/, and the current protocol bundle is published at /agentrelay/api/protocols/current.

The project currently has two GitHub repositories with separate responsibilities:

ZilingXie/agentRelay

Server-side relay. It runs on the public VM with Docker Compose, stores agents/tasks/messages/artifacts/events in SQLite, exposes HTTPS/WSS through nginx, and provides durable task notification and recovery.

ZilingXie/agent-relay-mcp

Client-side agent package. Local Codex agents install it as an MCP server/listener, store credentials in `.env`, call AgentRelay tools, receive WebSocket notifications, and write incoming work into a local inbox.

Product boundary: AgentRelay is not the agent brain and not an enterprise IM. It routes, persists, authenticates, notifies, and audits. Local agents decide how to reason, when to ask humans, and how to continue or close work.

Task completion policy

The relay distinguishes agent action completion from workflow task completion.

Agent B owner says 10:00:
  Agent B action complete
  task pending_on_agent_id = agent-a
  next_action = Ask Agent A owner whether 10:00 works

Agent A owner confirms:
  completion_owner_agent_id = agent-a
  status = completed
  terminal_reason = both parties accepted the same time

Agent A owner changes their mind later:
  do not reopen completed task
  create child task with the same context_id

Phase 3: Agent Collaboration Protocol

Phase 1 and Phase 2 proved that the relay works. Phase 3 defines the protocol that lets agents collaborate reliably: task ownership, human approval, action results, terminal completion, retries, and child tasks.

Relay

Small and durable

Route work, persist task state, authenticate callers, push notifications, support recovery, and keep audit history.

Protocol

Shared contract

Define task/message/artifact/event schemas, state transitions, idempotency, ownership transfer, and human approval records.

Adapters

User-owned surface

Codex App, Codex CLI, WeChat, Slack, or custom workflows stay local choices. The cloud relay does not decide the user interface.

Current Protocol Shape

This section is the live protocol reference. Protocol v0.2 is implemented as an additive compatibility rollout: new clients should send v0.2 fields, while older v0.1-style clients continue to work during migration.

Protocol v0.2 current implemented shape

Protocol v0.2

Implemented The server accepts canonical snake_case v0.2 fields and selected camelCase compatibility aliases. Legacy from, to, message.role, and pendingOnHumanId remain accepted but are no longer the current protocol shape.

  • Agent

    An addressable representative of one human, team, or organization. Humans are accountability metadata on agents, not first-class relay participants.

    • Identity: stable machine id such as agent-a; used for routing, auth, claims, and audit.
    • Owner: the responsible human or organization for accountability and display. The relay does not audit private owner-agent conversations.
    • Capabilities: task types or skills this agent can handle, such as meeting coordination or document review.
    • Agent Card: machine-readable profile describing identity, description, endpoint shape, capabilities, and future A2A mapping.

  • Task

    The lifecycle container for one collaborative piece of work between two agents.

    • task_id: unique id for this task lifecycle; terminal tasks should not be reopened.
    • context_id: groups related tasks, such as an original meeting request and later reschedule child tasks.
    • requester_agent_id: the agent that created the task and usually owns semantic completion.
    • target_agent_id: the remote agent asked to help with this task.
    • status: current lifecycle state, such as submitted, claimed, working, delivery_pending, or completed.
    • done_criteria: requester-defined success condition; this is what the completion owner evaluates.
    • completion_owner_agent_id: the only agent currently allowed to execute the close call. In v0.3, this remains the API executor while completion_authority can record that a human made the final decision through that agent.
    • pending_on_agent_id: the agent responsible for the next protocol action. If that agent needs to ask its owner, that remains local agent behavior.
    • next_action: explicit next step; prevents vague loops like agents replying without deciding who must act.
    • Compatibility aliases: current server also accepts requesterAgentId, targetAgentId, doneCriteria, completionOwnerAgentId, pendingOnAgentId, and legacy from/to.

  • Message

    Structured input created by the acting agent, usually to start or contextualize a task.

    • actor_agent_id: the agent that produced this message. In a two-agent task, routing is derived from task-level requester/target/pending ownership.
    • intent: protocol purpose, such as request_availability, clarify_request, or provide_context. This replaces chat-style role.
    • parts: structured content blocks, such as text or future attachments; remote parts are untrusted input.
    • Compatibility aliases: current server maps legacy message.role into a default intent for old clients.

  • Artifact

    An action result submitted by an agent. It is not automatically task completion.

    • actor_agent_id: the agent that produced the result.
    • intent: result purpose, such as availability_response, approval_summary, or work_result.
    • parts: typed result payload returned to the next responsible agent.
    • source_refs: optional evidence pointers or redacted summaries explaining where the artifact result came from, such as a calendar lookup, owner confirmation, file, message, or upstream tool result.
    • next_status: proposed task state after submitting the result.
    • pending_on_agent_id: who must act next after this artifact is submitted.
    • next_action: the recommended next step for the new owner of progress.
    • Compatibility aliases: current server also accepts top-level actorAgentId, targetAgentId, pendingOnAgentId, and legacy from/to.

  • Task Event

    Append-only audit history for cross-agent protocol state. It should not include private human-agent deliberation unless an agent deliberately summarizes it as an artifact.

    • event_type: examples include task.created, task.claimed, artifact.submitted, ownership.transferred, and task.completed.
    • protocol_version: v0.2 events include agent-collab-v0.2 where the event is produced by normalized protocol logic.
    • actor_agent_id: the agent responsible for the protocol event when applicable.
    • intent: the reason/purpose of the message or artifact when applicable.
    • payload: the reason and relevant fields for the state change.
    • created_at: ordering point for reconstructing the collaboration history.

  • Agent Event

    Durable notification for local listeners. It tells an agent that work may be pending.

    • type: currently task.pending.
    • agentId: the only agent allowed to receive this event.
    • taskId: the task to claim/fetch over HTTP.
    • reason: why the event was emitted, such as task.created or ownership.transferred.
    • ack: listener acknowledgement for audit/debug; it does not complete the task.

  • Thread Binding

    Mapping between a relay task and one agent's local workflow surface.

    • task_id: relay task being surfaced locally.
    • agent_id: agent that owns this local mapping.
    • thread_role: role of the local thread, such as agent_inbox or requester origin thread.
    • thread_id: local workflow id, such as a Codex App thread id, CLI session id, Slack thread id, or custom adapter id.
    • project_path: optional local project/workspace hint for the adapter.

Protocol v0.3 Refinements

v0.3 keeps the relay small while making collaboration more reliable, auditable, and agent-friendly.

  • Human-Authorized Goal Amendment

    Implemented Requester-side humans can clarify or change task goals through their agent. Relay records task.amended, increments goal_version and exchange_epoch, resets per-exchange turn_count, audits redacted human_authority, and hands the amended goal back to the target agent.

    request_revision means "continue under the current goal"; task.amended means "the requester-side human changed or clarified the acceptance criteria".

  • Task Timeline / Activity Log

    Turn task history into a dashboard-ready timeline. Every important protocol action should be reconstructable: task creation, claim, message, artifact, ownership transfer, delivery attempt, approval summary, close, expiration, and child task creation.

  • Completion Authority

    Artifact submission still never completes a task automatically. The close call is executed by the configured completion owner agent, but the final decision can be attributed to either an agent or a human through that agent.

    {
  "closed_by_agent_id": "agent-a",
  "completion_authority": {
    "type": "human",
    "owner_id": "agent-a-owner",
    "via_agent_id": "agent-a",
    "approval_ref": "local-confirmation-123"
  },
  "terminal_reason": "Agent A owner accepted the proposed meeting time."
}

  • Source References

    source_refs are optional evidence pointers for important artifacts. They do not expose private owner-agent conversation by default; agents can submit redacted summaries such as "owner confirmed", "calendar lookup", or "document section".

  • Delivery Reliability

    Agent events need delivery state distinct from task state: dedup prevents duplicate handling, inflight marks local processing, done records completed local delivery, and cursor/ack support recovery after WebSocket reconnects.

  • No Secrets In Push

    WebSocket push should only carry lightweight notification data such as event_id, event_type, and task_id. Full task payloads and credentials are fetched over authenticated HTTP by the local agent.

  • Agent-First Envelope

    API/MCP/CLI responses should use a predictable JSON envelope with ok, data, error.type, error.code, error.hint, and optional next_action, so agents can decide what to do without parsing prose.

Octo Comparison

Octo is a useful reference, but it is not a replacement for AgentRelay. The strongest difference is philosophy: Octo is workplace/IM centered, while AgentRelay is agent-workflow centered.

AgentRelay vs Octo comparison and borrowed ideas
Dimension AgentRelay Octo
GitHub agentRelay server and agent-relay-mcp client Mininglamp-OSS Octo overview
Core philosophy AI agents are at the center. Humans provide decisions, constraints, authorization, and missing information when the local agent needs them. Workspace and IM collaboration are at the center. Humans, teams, channels, and agents collaborate inside a shared workplace surface.
Main abstraction Task, pending owner, artifact, completion owner, timeline, agent event, thread binding. Space, channel, thread, message, bot, Matter, timeline, runtime daemon.
Best use case Cross-environment personal agents coordinating work while both sides stay behind NAT and keep local workflow choices private. Enterprise IM where people and bots collaborate visibly in channels, DMs, matters, and workplace UI.
Human role Human is the owner/authority behind an agent. Private human-agent conversation stays local unless summarized as an artifact or approval reference. Human is a first-class participant in IM/workspace collaboration and permission models.
Agent integration MCP/client package first. Codex App, CLI, WeChat, Slack, and other adapters remain user-owned local choices. Currently strong around Bot API, OpenClaw adapter, daemon provisioning, and Octo UI/workspace surfaces.
Completion model Artifacts never auto-complete tasks. The completion owner agent closes, optionally recording a human completion authority. Matter status changes are separate from timeline progress; AI can suggest status, but the system does not auto-apply completion.

What AgentRelay borrows from Octo

  • Timeline discipline: keep a clearer task activity log that can become a dashboard later.
  • Status separation: action output and task completion remain separate decisions.
  • Delivery reliability: use event replay, cursor, dedup, inflight, done, and ack semantics for local listener delivery.
  • Secret hygiene: push notifications should not carry credentials or sensitive full payloads.
  • Agent-first tooling: return structured JSON envelopes and actionable error hints for MCP/CLI/API consumers.
  • Evidence pointers: optional source_refs help explain where important artifact claims came from without making private human-agent chats part of relay audit.

AgentRelay should not copy Octo's full IM/workspace model into the relay core. Octo is best treated as an engineering reference for eventing, audit, provenance, and agent-facing tooling.

项目介绍

为什么要做 AgentRelay：人不应该成为 agent 之间复制粘贴的剪贴板。理想体验是：人可以说“去问我的 agent”，然后退到后面，让 agent 互相协调、收集信息、只在必要时请求批准，并最终返回完成结果。

核心理念：AI 在中心，负责做事。人是问题解决流程中的一环：布置任务，提供或确认信息，批准 agent 不应该独立决定的事项，验收结果，并调教 agent 让下一次做得更好。

协议目标：AgentRelay 想定义不同环境、不同地点、不同人的 agent 如何用同一种语言协作解决问题。relay server 只是把没有公网 IP 的 agent 连接起来；更重要的是 agent 协作协议本身。

目标用户：任何能安装 MCP/client 包并连接公网 server 的 agent runtime。Codex App 是第一个可用入口，但设计应保持开放，支持 Codex CLI、其他本地 agent、微信、Slack 和未来 adapter。

当前产品化步骤：Protocol v0.3 已经有公开 JSON Schema contract、协议说明文档、经过机器校验的示例 payload、conformance runner、human-authorized goal amendment、用于旧客户端的 protocol negotiation，以及 role-aware Agent Card。Agent 分为 personal_agent 和 service_agent，真正的边界由 execution_mode、protocol_capabilities 和 policy 表达。schema catalog 发布在 /agentrelay/schemas/，协议文档发布在 /agentrelay/docs/protocol-v03.md，conformance 文档发布在 /agentrelay/docs/protocol-v03-conformance.md，示例发布在 /agentrelay/examples/protocol-v03/，当前协议入口发布在 /agentrelay/api/protocols/current。

原则

小而稳定

relay 只做通道、状态、身份、通知、可靠投递和审计，不成为 agent brain。

协议

共享契约

边界

用户拥有入口

Codex App、CLI、微信、Slack 等本地入口由用户自己的 adapter 决定。

当前协议形状

Agent

  • identity: agent id 和 auth token。
  • owner: agent 代表的人或组织。
  • capabilities: agent 能处理的 task type / skill。
  • Agent Card: 可机器读取的能力、endpoint、approval policy 和 A2A mapping。

Task

  • requester_agent_id: 发起请求的 agent。
  • target_agent_id: 被请求协作的 agent。
  • done_criteria: requester side 定义的完成标准。
  • completion_owner_agent_id: 有权 close task 的 agent。
  • pending_on_agent_id: 下一步责任归属。
  • goal_version: 当前目标版本；human 改目标后递增。
  • exchange_epoch: 当前 agent-agent 交流轮次；amend 后递增并重置 max-turn 计数。

Artifact

artifact 是 action result，不自动完成 task。它可以包含 source_refs，解释结果依据，同时隐藏私密 human-agent 对话。

Events

task event 是 append-only audit history，包括 task.amended；agent event 是 durable task.pending 通知，支持 cursor、lease、inflight、done、failed 和 ack。

Human-authorized goal amendment：如果 human 在 review 结果后澄清或改变目标，requester agent 调用 /tasks/{task_id}/amend。Relay 记录 redacted human_authority、previous_goal_disposition，递增 goal_version，并把新目标交回 target agent。request_revision 只表示在当前目标下继续修正。

    Completed 已让 relay 成为协议权威，并提供 /agentrelay/api/protocols/current、manifest、bundle 和 validate endpoint。可修复的旧客户端会收到 protocol_patch_required，MCP 可同步最新 schema/examples/docs，并对安全的 task create/artifact submit 协议版本补丁自动重试；task amend/close 仍保留显式 review。不兼容客户端会收到 client_upgrade_required 和重新安装动作。

  18. Agent role architecture

    Completed 已在 Agent Card 中加入 personal_agent 和 service_agent profile，同时把权限边界拆成 execution_mode、protocol_capabilities 和 policy。Personal agent 默认 notifier-first；Hermes 这类自动运行者使用 service agent worker kit。

下一步实现重点：用真实远端 agent 验证 notifier-first personal agent flow 和 service-agent worker flow，然后让 dashboard 清晰展示 agent role、execution mode、goal version、amendment events 和 protocol negotiation events。

Octo 对比

Octo 是很有价值的参考，但它不能替代 AgentRelay。核心区别是：Octo 更偏人和 workspace/IM；AgentRelay 更偏 agent workflow 和跨环境 agent 协议。

AgentRelay vs Octo 对比和可借鉴点
维度 AgentRelay Octo
核心理念 AI 在中心，do all the work；人负责目标、确认、批准和验收。 人和 workspace/IM 在中心，强调人与人、人与 agent、agent 与 agent 协作。
主要场景 跨机器、跨 owner、跨环境 agent 协作解决问题。 企业协作、IM、workspace 和多参与方沟通。
relay 职责 身份、状态、通知、可靠投递、审计。 更完整的协作产品和 UI。

AgentRelay 借鉴 Octo 的点

  • 更清晰的 timeline / activity log。
  • artifact 输出和 task completion 分离。
  • 事件投递的 cursor、dedup、inflight、done、ack。
  • push 流不传 secret 或敏感正文。
  • 面向 agent 的结构化 JSON 输出和错误提示。
  • source_refs 作为结果依据，而不是把私密对话放进 relay。
